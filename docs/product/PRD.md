# MailMind 产品需求说明书（PRD）

**版本**：v1.2  
**最后更新**：2026-06-16  
**产品阶段**：MVP  

---

## 1. 项目概述

### 1.1 产品名称与类型

正式产品名：MailMind

产品类型 / 副标题：AI Email Copilot

仓库名：mailmind-ai-email-copilot

### 1.2 产品定位

MailMind 是一款面向高频邮件用户的 **AI Email Copilot**。产品通过接入用户 Gmail 邮箱，对每日邮件进行自动读取、理解、聚合与分析，并生成结构化的 AI 邮件日报，帮助用户快速识别当日必须处理、建议查看和可忽略的邮件内容。

本产品的核心不是替代传统邮箱客户端，而是为用户提供一个 **AI 驱动的邮件决策入口**，使用户能够从"逐封浏览邮件"转变为"基于 AI 建议进行确认和处理"。

### 1.3 产品核心价值

当前邮件系统普遍存在以下问题：

1. 邮件数量较多，用户难以快速判断优先级
2. 重要邮件容易被广告、通知类邮件淹没
3. 用户需要花费大量时间阅读、判断和处理邮件
4. 邮件工具大多以收件箱列表为中心，缺少面向行动的决策辅助
5. 用户每天难以快速掌握邮件整体情况和待办事项

MailMind 通过 AI 对邮件进行理解和判断，将邮件信息转化为**可执行的行动建议**，从而提升用户处理邮件的效率和确定性。

### 1.4 产品定性说明

当前阶段为**个人自用与本地自托管工具**，不作为 SaaS 产品对外运营。MVP 阶段暂不建设多租户隔离和商业合规体系，但仍需遵循权限审慎、Token 加密存储、敏感信息不入日志、用户可清除本地数据等基础安全原则。

## 2. 产品目标

### 2.1 MVP 阶段目标

MVP 阶段重点验证产品的核心价值，即：

> 用户完成 Gmail 授权后，系统能够自动获取当日邮件，并生成一份面向行动的 AI 邮件日报。

MVP 阶段不追求完整邮箱客户端能力，而是优先将"每日 AI 邮件决策"做到稳定、清晰、可用。

### 2.2 MVP 核心能力

MVP 阶段仅聚焦以下核心能力：

1. Gmail 邮箱 OAuth 授权接入
2. 获取当日邮件数据
3. 对邮件内容进行 AI 分析
4. 生成每日 AI 邮件决策看板（Daily Digest）
5. 支持用户查看关联邮件详情
6. 提供基础邮件操作能力（标记已读、未读、打开原始邮件）
7. 支持手动触发日报生成，以及可配置的定时自动生成

### 2.3 非 MVP 阶段能力

以下能力不作为 MVP 首要目标，在后续版本迭代：

1. Outlook 邮箱接入
2. IMAP 邮箱接入（QQ 邮箱、163 邮箱、企业邮箱等）
3. 多邮箱统一聚合收件箱
4. 邮件 AI 自动回复
5. 邮件线程级深度总结
6. 自动归档、自动打标签
7. 桌面客户端（Electron）
8. 浏览器插件
9. 移动端 App

## 3. 目标用户

### 3.1 主要用户群体

当前阶段为个人自用产品，主要面向以下场景：

1. 高频邮件用户（每日邮件量 ≥ 20 封）
2. 开发者、产品经理、运营人员、商务人员等知识工作者
3. 需要同时关注大量通知、任务、协作邮件的用户
4. 求职、招聘、项目协作等场景下需要快速筛选邮件的用户

### 3.2 用户核心痛点

1. 每天需要处理大量邮件，但无法快速判断哪些邮件真正重要
2. 传统邮箱以时间顺序展示邮件，缺少智能优先级判断
3. 用户需要反复打开邮件、阅读正文、判断是否需要处理
4. 通知类、营销类、系统类邮件干扰较多
5. 用户希望每天能够快速了解"今天有哪些邮件需要我处理"

## 4. 产品设计原则

### 4.1 AI 决策优先

产品首屏不以传统邮件列表为核心，而以 **AI 生成的每日邮件决策看板**为核心。系统应优先向用户展示"今天需要处理什么"，而不是简单展示"今天收到了什么"。

### 4.2 行动导向

AI 输出不应停留在摘要、分类等描述性结果，而应转化为**可执行建议**，例如：

- 建议今日回复
- 建议今日查看
- 可忽略
- 需要在指定时间前处理
- 建议稍后跟进

### 4.3 用户确认机制

AI 不直接替用户执行高风险操作。对于回复、归档、忽略等行为，系统应提供建议，由**用户最终确认**执行。

### 4.4 邮箱操作能力作为支撑层

邮件列表、邮件详情、已读未读状态等能力是产品的**基础支撑能力**，而非产品主入口。它们服务于 AI 决策看板中的具体操作场景。

## 5. 核心用户流程

### 5.1 首次使用流程

1. 用户进入系统
2. 系统引导用户连接 Gmail 邮箱
3. 用户完成 Google OAuth 授权
4. 系统获取授权范围内的邮件数据
5. 系统同步当日邮件
6. AI 对当日邮件进行分析
7. 系统生成今日 AI 邮件决策看板

### 5.2 日常使用流程

1. 用户打开系统
2. 首页展示今日 AI 邮件决策看板（如当日日报已存在则直接展示，否则提示用户生成）
3. 用户查看今日邮件整体摘要
4. 用户优先处理"必须处理"邮件
5. 用户按需查看"建议查看"邮件
6. 用户确认或忽略 AI 标记为低优先级的邮件
7. 用户可进入邮件详情查看原文并执行操作

### 5.3 Daily Digest 生成与增量更新机制

#### 5.3.1 问题背景

Daily Digest 是基于某一时间点生成的 AI 邮件决策快照。如果用户在日报生成后继续收到新邮件，则当前日报可能无法覆盖最新邮件。系统必须明确展示日报的生成时间、覆盖范围和新邮件状态，避免用户误以为日报已包含全天所有邮件。

#### 5.3.2 日报触发方式

MVP 阶段采用以下触发方式：

- **手动触发**：用户点击"生成今日日报"按钮手动触发
- **定时任务**：用户可在设置中配置定时自动生成时间（默认每天 08:00），系统在后台自动完成同步和生成

日报生成后不自动覆盖，新邮件到达后只提示用户，用户点击刷新后才重新生成。

#### 5.3.3 日报状态定义

每份 Daily Digest 必须包含以下状态字段：

```json
{
  "digest_id": "string",
  "date": "2026-06-16",
  "generated_at": "2026-06-16T09:00:00+09:00",
  "coverage_start": "2026-06-16T00:00:00+09:00",
  "coverage_end": "2026-06-16T09:00:00+09:00",
  "mail_count": 24,
  "new_mail_count_after_digest": 0,
  "status": "fresh"
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `generated_at` | 日报生成时间 |
| `coverage_start` | 日报分析起始时间 |
| `coverage_end` | 日报分析截止时间 |
| `new_mail_count_after_digest` | 日报生成后新增且尚未合并分析的邮件数量 |
| `status` | 当前日报状态 |

#### 5.3.4 日报状态类型

| 状态 | 含义 | 前端表现 |
|---|---|---|
| `fresh` | 当前日报覆盖最新邮件 | 显示"已是最新" |
| `stale` | 日报生成后存在新邮件 | 显示"有 N 封新邮件未分析" |
| `refreshing` | 正在同步并重新生成日报 | 显示生成进度 |
| `failed` | 日报生成失败 | 显示错误与重试按钮 |

#### 5.3.5 新邮件检测与展示

用户打开系统首页时，后端检查 Gmail 中是否存在日报生成后收到的新邮件。

若存在新邮件，首页顶部显示提示：

> 今日日报生成于 09:00。此后收到 3 封新邮件，尚未纳入 AI 分析。

提供操作按钮：

- **刷新日报**：重新同步并生成包含新邮件的日报
- **查看新邮件**：仅查看新邮件列表，附带简要 AI 判断
- **稍后处理**：保留当前日报，忽略提示

若用户不刷新，原日报继续保留，同时标记为"可能不是最新"。

#### 5.3.6 AI 生成失败降级策略

当 AI 日报生成失败时，系统不应展示空白页面，而应：

1. 展示原始邮件列表（按时间倒序）
2. 提供基础筛选能力（发件人、已读/未读）
3. 顶部显示错误提示及重试按钮
4. 保留上次生成成功的日报（若存在），提示用户当前展示为历史日报

#### 5.3.7 后续增强方向

后续版本可增加以下能力：

- Gmail Push Notification 实时感知新邮件
- 增量 AI 分析新邮件，按优先级插入当前日报对应区域
- 自动将高优先级新邮件插入首页看板
- 用户自定义刷新频率

## 6. 功能需求

### 6.1 Gmail 接入模块

#### 6.1.1 功能说明

系统支持用户通过 Google OAuth 授权接入 Gmail 邮箱，并在授权范围内读取邮件数据和执行基础邮件操作。

#### 6.1.2 功能要求

1. 支持 Google OAuth 2.0 登录授权
2. 支持保存用户授权信息（Access Token / Refresh Token）
3. 支持自动刷新 Access Token
4. Refresh Token 失效时提示用户重新授权
5. 支持获取当日 Gmail 邮件列表
6. 支持读取邮件正文、主题、发件人、收件人、时间等信息
7. 支持识别邮件已读、未读状态
8. 支持打开邮件原始 Gmail 链接
9. 支持用户主动断开 Gmail 授权
10. MVP 阶段仅支持 Gmail，不支持 Outlook 和 IMAP

#### 6.1.3 权限要求

MVP 采用自用完整体验权限集，申请支持当前自用体验所需的 Gmail 权限范围：

| 权限范围 | 用途 |
|---|---|
| `gmail.readonly` | 读取邮件列表和正文 |
| `gmail.modify` | 标记已读、未读状态 |

其中 `gmail.modify` 用于支持 Gmail 已读 / 未读状态真实同步，不应表述为严格意义上的只读最小权限。该 scope 属于受限 Gmail 权限，未来如果公开发布或 SaaS 化，需要重新评估 Google OAuth 审核、安全评估和权限最小化策略。

系统不申请发送邮件权限（`gmail.send`），MVP 阶段不支持在系统内发送邮件。

#### 6.1.4 可扩展性要求

Gmail 接入层应采用 **Provider Adapter 模式**封装，便于后续支持 Outlook（Microsoft Graph API）和 IMAP 协议，上层业务逻辑不感知具体邮箱协议差异。

---

### 6.2 今日 AI 邮件决策看板

#### 6.2.1 功能说明

今日 AI 邮件决策看板是产品**首页和核心功能**。系统基于当日邮件内容生成结构化日报，帮助用户快速判断邮件优先级和处理动作。

#### 6.2.2 看板内容区域

| 区域 | 说明 |
|---|---|
| 今日邮件概览 | 邮件总数、优先级分布、一句话总结 |
| 必须处理 | 高优先级、需要今日回复或处理的邮件 |
| 建议查看 | 中优先级、建议今日阅读的邮件 |
| 可忽略 | 低优先级、通知类、营销类邮件 |
| 今日待办事项 | 从邮件中提取的 Action Items |
| 风险提醒 | 未回复的重要邮件、临近截止日期的事项 |

#### 6.2.3 今日邮件概览示例

> 今日共收到 28 封邮件，其中 3 封建议优先处理，5 封建议查看，其余多为通知和营销邮件。

---

### 6.3 邮件行动建议模块

#### 6.3.1 功能说明

系统将邮件分析结果转化为面向用户的**行动建议**，而不是仅展示分类结果。

#### 6.3.2 行动类型定义

| 行动类型 | 含义 |
|---|---|
| `reply_today` | 建议今日回复 |
| `review_today` | 建议今日查看 |
| `handle_before_deadline` | 建议在截止时间前处理 |
| `ignore` | 可忽略 |
| `archive_candidate` | 建议归档 |
| `follow_up_later` | 建议稍后跟进 |
| `no_action_required` | 无需操作 |

#### 6.3.3 每条建议包含的信息

```json
{
  "email_id": "gmail_xxx",
  "subject": "Project timeline update",
  "sender": "alice@example.com",
  "suggested_action": "review_today",
  "priority": "high",
  "reason": "邮件包含项目排期变更，可能影响当前开发计划。",
  "deadline": null,
  "confidence": 0.86
}
```

#### 6.3.4 置信度展示规则

| 置信度范围 | 前端展示策略 |
|---|---|
| `≥ 0.8` | 正常展示 AI 建议动作 |
| `0.6 – 0.8` | 展示建议动作，附注"AI 不确定，建议自行判断" |
| `< 0.6` | 不显示具体建议动作，强制归入"建议查看"区域 |

### 6.4 邮件详情模块

#### 6.4.1 功能说明

用户从 AI 看板点击某封邮件后，进入邮件详情页面查看原始邮件内容和 AI 分析结果。

#### 6.4.2 展示内容

1. 邮件主题
2. 发件人
3. 收件人
4. 收件时间
5. 邮件正文
6. AI 摘要
7. AI 建议动作及原因
8. 关联待办事项
9. 基础操作按钮

#### 6.4.3 基础操作

MVP 阶段支持：

1. 标记为已读
2. 标记为未读
3. 打开 Gmail 原始邮件链接
4. 返回今日看板

邮件回复能力作为后续版本增强，不在 MVP 范围内。

### 6.5 邮件基础列表模块

#### 6.5.1 功能说明

邮件列表作为**辅助功能**存在，支撑用户查看完整邮件数据，不作为产品首页主入口。

#### 6.5.2 功能要求

1. 展示当日邮件列表
2. 支持按时间倒序排列
3. 支持按 AI 优先级筛选（高 / 中 / 低）
4. 支持点击进入邮件详情
5. 支持展示邮件已读状态

## 7. AI 输出要求

### 7.1 结构化输出

AI 输出必须采用**结构化 JSON 格式**，便于系统解析、存储和前端展示。

### 7.2 可执行性要求

AI 输出必须包含明确的建议动作，不允许只输出描述性摘要。

❌ 错误示例：

```json
{
  "summary": "这是一封关于项目进度的邮件。"
}
```

✅ 正确示例：

```json
{
  "summary": "该邮件通知项目交付时间提前。",
  "suggested_action": "review_today",
  "reason": "项目排期变化可能影响当前工作安排。",
  "priority": "high",
  "confidence": 0.88
}
```

### 7.3 邮件分类维度说明

邮件分析应使用**两个独立维度**，避免混用：

| 维度 | 字段名 | 可选值 |
|---|---|---|
| 邮件类型 | `category` | `work` / `notification` / `marketing` / `social` / `other` |
| 优先级 | `priority` | `high` / `medium` / `low` |

两个维度独立赋值，例如一封重要的工作邮件为 `category: work, priority: high`，一封工作类通知邮件为 `category: work, priority: low`。

### 7.4 风险控制

对于不确定的邮件，AI 不应强行给出高优先级判断，置信度低于 0.6 时应标记为"建议查看"，由用户自行判断。

## 8. 数据范围

### 8.1 MVP 数据范围

MVP 阶段仅处理：

1. Gmail 邮箱
2. 当日收到的邮件（以用户本地时区为准）
3. 邮件正文文本内容
4. 基础邮件元数据（主题、发件人、收件人、时间、已读状态）

### 8.2 暂不处理的数据

1. 历史邮件深度分析
2. 附件内容解析
3. 多邮箱合并
4. 复杂邮件线程推理
5. 自动发送邮件

## 9. 非功能需求

### 9.1 性能要求

1. 首页看板加载时间应控制在 3 秒以内（日报已生成时）
2. 当日邮件同步应在合理时间内完成，前端展示同步进度
3. AI 日报生成过程异步执行，前端展示生成状态
4. 已生成日报应缓存，避免重复消耗模型调用资源

### 9.2 安全要求

1. 不存储用户 Gmail 密码
2. 使用 OAuth 2.0 授权
3. Access Token 和 Refresh Token 加密存储
4. 用户可主动断开 Gmail 授权并清除本地 Token
5. 系统不得在未经用户确认的情况下发送邮件
6. Gmail 权限采用自用完整体验权限集：`gmail.readonly + gmail.modify`，未来公开发布或 SaaS 化时重新评估权限最小化策略

### 9.3 可扩展性要求

1. 邮箱接入层采用 Provider Adapter 模式，便于后续支持 Outlook 和 IMAP
2. AI 分析模块独立封装，支持替换不同大模型（Claude / OpenAI 等）
3. Daily Digest 生成逻辑与前端展示解耦
4. 后端 API 设计支持未来桌面客户端复用

### 9.4 可靠性要求

1. 邮件同步失败时支持自动重试
2. AI 调用失败时展示降级页面（见 5.3.6）
3. Access Token 过期时自动刷新
4. Refresh Token 失效时提示用户重新授权
5. 系统记录关键任务执行状态，便于排查问题

## 10. 产品边界

MVP 阶段明确不做以下内容：

1. 不做完整邮箱客户端
2. 不做邮件营销系统
3. 不做企业 CRM
4. 不做复杂自动回复 Agent
5. 不做多邮箱统一收件箱
6. 不做移动端 App
7. 不做桌面客户端
8. 不做附件内容深度解析
9. 不做自动发送邮件
10. 不做 SaaS 多租户能力

## 11. 成功标准

### 11.1 产品成功标准

1. 用户能够完成 Gmail 授权
2. 系统能够成功获取当日邮件
3. 系统能够生成结构化 AI 邮件日报
4. 用户能够通过日报快速识别当日需要处理的邮件
5. 用户能够从 AI 看板进入邮件详情并完成基础操作
6. AI 优先级判断准确率达到用户主观满意（以个人使用反馈为准）
7. AI 回复草稿（后续版本）采纳率 ≥ 50%

### 11.2 工程成功标准

1. Gmail 接入逻辑与业务逻辑解耦，符合 Provider Adapter 模式
2. AI 分析模块可独立测试和替换
3. 邮件数据结构统一，上层业务不感知邮箱差异
4. Daily Digest 结果可存储、复用和刷新
5. 系统具备后续扩展 Outlook、IMAP、多邮箱聚合的架构基础

## 12. MVP 最终定义

MailMind 的 MVP 定义为：

> 一个基于 Gmail 的 AI 邮件决策系统。用户授权 Gmail 后，系统自动读取当日邮件，并生成以"必须处理、建议查看、可忽略、待办事项和风险提醒"为核心结构的每日 AI 邮件日报。用户以日报看板为主入口，根据 AI 建议进入具体邮件并执行基础操作。日报支持手动触发和定时自动生成，新邮件到达后通过顶部提示告知用户，由用户决定是否刷新日报。

## 13. AI Provider 配置路线图

### 13.1 MVP 阶段（当前）

MVP 使用 `.env` 环境变量配置 AI Provider，通过 `LLMClient` 接口抽象实现 provider 切换能力：

1. 系统通过 `.env` 中的 `LLM_PROVIDER` 和 `LLM_API_KEY` 配置唯一 AI Provider；
2. `LLMClient` 工厂根据环境变量初始化对应的 provider 实例（Anthropic / OpenAI 等）；
3. 所有 AI 调用通过 `ai_runs` 表的 `model_provider` 和 `model_name` 字段记录；
4. **用户不可配置**，无需 UI，无需数据库存储。

### 13.2 V1 阶段：用户可配置 AI Provider

V1 引入用户级 AI Provider 配置能力：

1. 新增 `ai_provider_configs` 表，存储用户配置的 provider（provider type、加密 API Key、base_url 等）；
2. 用户可通过 `/settings/ai` 页面添加、编辑、测试、启用/禁用 provider；
3. 支持 OpenAI、Anthropic Claude、Google Gemini、Azure OpenAI、Ollama / 本地模型；
4. API Key 加密存储（复用 `APP_ENCRYPTION_KEY` + `encryption_key_version` 模式）；
5. `ai_runs` 增加 `ai_provider_config_id` 外键，追踪每次 AI 调用使用的 provider；
6. 前端仅返回 masked key，禁止明文返回；
7. 自定义 `base_url` 必须通过 SSRF 防护校验（禁止内网 IP / metadata 地址）。

### 13.3 V2 阶段：Model Profile 与 Agent 配置

V2 引入多模型切换和 Coding Agent 配置：

1. 新增 `ai_model_profiles` 表，支持用户为不同任务（digest / reply_draft / risk_detection）配置不同模型；
2. 支持 Codex、Claude Code、Coding Plan 等 Agent 工具的凭据配置（与 LLM provider 分离管理）；
3. 支持 fallback chain：主 provider 失败时自动切换到备用 provider；
4. AI 使用量追踪与成本估算。

### 13.4 分类原则

以下分类在本项目中严格区分：

| 分类 | 示例 | 存储位置 | 说明 |
|------|------|---------|------|
| **LLM Provider** | OpenAI、Anthropic、Google、Ollama | `ai_provider_configs` | AI 推理服务 |
| **Coding Agent** | Codex、Claude Code | V2 单独表或 config_json | 开发工具，不属于 LLM provider |
| **Email Provider** | Gmail、Outlook、IMAP | `mailboxes.provider` | 邮箱服务 |

Codex 和 Claude Code **不得**作为 `ai_provider_configs` 的 provider 类型值。它们是消费 LLM 的工具层，不是 LLM 推理服务本身。
