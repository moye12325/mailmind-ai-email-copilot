# MailMind Demo Script

**Duration:** 5–8 minutes  
**Audience:** Technical interviewers, portfolio reviewers, open-source contributors  
**Prerequisites:** MailMind running locally (backend + frontend + worker + Redis + PostgreSQL)

---

## Demo Flow

### 1. Opening — The Problem (30s)

> "Email overload is a daily reality. MailMind is an AI email copilot that turns your inbox into a prioritized decision queue. Instead of scanning 50 emails, you get a dashboard of what matters, ranked by urgency, with suggested actions."

### 2. Dashboard — All Mailboxes Digest (90s)

**Navigate to:** `http://localhost:3000/dashboard`

**What to show:**
- The Daily Digest dashboard is the primary view
- Today's emails are pre-classified into sections: **Urgent**, **Review**, **Ignore**, **To-Do**, **Risk**
- Each item shows: sender, subject, priority badge, AI confidence score, suggested action
- The overview section at top summarizes total mail count and per-mailbox highlights

**Talking points:**
- "This digest was generated by an LLM that analyzed all emails received since midnight in my timezone"
- "Items are scored by confidence — the AI is transparent about what it's certain about vs. what needs review"
- "The dashboard is the default landing page — designed as a decision interface, not another inbox"

### 3. Theme Showcase (30s)

**Navigate to:** Settings or use the theme switcher in the header

**What to show:**
- Toggle between themes: Neon Cyber → Glass Aurora → Soft Clay → Dense Minimal
- Toggle light/dark mode within a theme
- Show the pre-hydration flash prevention (themes load instantly, no FOUC)

**Talking points:**
- "6 theme presets, each with full light and dark mode support"
- "Theme state persists in localStorage and applies before React hydrates — no flash"
- "Every theme was Playwright-verified across 12 combinations"

### 4. Multi-Mailbox — Switch Between Digests (60s)

**Navigate to:** Dashboard → mailbox selector or sidebar

**What to show:**
- Switch from "All Mailboxes" to a single Gmail mailbox
- The digest re-scopes to show only that mailbox's emails
- Switch to an IMAP mailbox — different email source, same digest interface
- Show that each mailbox has independent sync status

**Talking points:**
- "MailMind supports Gmail and IMAP out of the box, with Outlook planned"
- "The Provider abstraction means adding a new email source is just implementing the MailboxProvider protocol"
- "Each mailbox has its own credentials, sync schedule, and digest scope"

### 5. Mailbox Management (45s)

**Navigate to:** `http://localhost:3000/settings/mailboxes`

**What to show:**
- List of connected mailboxes with provider type, email address, status, last sync time
- Show the connect flow for Gmail (OAuth) and IMAP (host/port/credentials)
- Show mailbox status indicators (active, reauth_required, disconnected)

**Talking points:**
- "Gmail uses OAuth2 — the user authorizes once and MailMind handles token refresh automatically"
- "IMAP supports any standard server — just provide host, port, and credentials"
- "Credentials are encrypted at rest using the application's encryption key"

### 6. Trigger Sync & Job Monitor (60s)

**Navigate to:** Settings → Mailboxes → click Sync on a mailbox

**What to show:**
- Click the sync button
- Show the job appears in the job list (if visible) or show the status updates
- The sync runs asynchronously via Celery — the UI stays responsive
- Emails appear in the email list after sync completes

**Talking points:**
- "Sync jobs use a commit-then-dispatch pattern — the job is committed to PostgreSQL before being sent to Celery"
- "If Celery dispatch fails, the job is marked dispatch_failed and can be retried"
- "Redis distributed locks prevent concurrent syncs on the same mailbox"
- "Stale jobs are automatically detected and replaced"

### 7. Email Browser (30s)

**Navigate to:** `http://localhost:3000/emails`

**What to show:**
- Emails listed by received time, filterable by mailbox
- Show search functionality
- Click into an email to see full details
- Show the "new emails" indicator for emails received after digest generation

**Talking points:**
- "Emails are the source of truth — they're synced from providers and stored locally"
- "The digest references emails but adds AI analysis on top"

### 8. Action History (30s)

**Navigate to:** `http://localhost:3000/actions`

**What to show:**
- Action history showing user interactions: viewed digest, opened email, marked as read, dismissed item
- Each action has timestamps and source context

**Talking points:**
- "Every interaction is tracked as a UserAction — this enables analytics and undo functionality"
- "Actions capture before/after state snapshots for debugging"

### 9. Architecture Highlights (60s)

**Reference:** [Architecture Diagrams](../architecture/DIAGRAMS.md)

**Key patterns to explain:**

| Pattern | Why it matters |
|---------|---------------|
| **Provider Protocol** | Adding a new email provider (Outlook, Yahoo) requires implementing one interface |
| **Commit-then-dispatch** | No lost jobs even if the process crashes between DB write and Celery dispatch |
| **Versioned digests** | New digest versions preserve history — users can compare or revert |
| **Idempotent job keys** | Partial unique index prevents duplicate in-flight jobs for the same operation |
| **Distributed locking** | Redis-based locks with TTL prevent concurrent syncs and handle worker crashes |
| **Defensive LLM parsing** | The AI output parser handles malformed JSON, aliased enums, and missing fields gracefully |

### 10. Closing (15s)

> "MailMind is a local-first system — your emails never leave your machine unless you configure an external AI provider. The architecture is designed for extensibility: new providers, new AI models, new digest scopes. The roadmap includes Outlook integration, AI settings UI, and open-source readiness."

## Demo Tips

### Before the demo
- Ensure all services are running: `docker-compose up -d` for infra, backend, worker, frontend
- Have at least one Gmail and one IMAP mailbox connected
- Trigger a sync so there are recent emails to show
- Generate a digest before the demo so the dashboard has content

### During the demo
- Start with the dashboard — it's the most visually impressive
- Don't spend too long on any single page — move quickly
- If something is slow (API call), use the time to explain the architecture
- Have the architecture diagrams open in a separate tab for reference

### Backup scenarios
- If the AI digest is empty: explain the prompt structure and show the AIRun record
- If a sync fails: show the error state and explain how the retry mechanism works
- If the UI is slow: explain that this is a local dev build, production would use compiled assets
