# MailMind v0.4.0 Job Experience

- Release name: MailMind v0.4.0 Job Experience
- Base version: v0.3.0-async-redesign
- Merged branch: feat/v04-job-experience
- Manual QA: passed, confirmed by user

## Summary

v0.4 connects the v0.3 Jobs API, async sync, async digest, and retry capabilities to the frontend job experience. Users can now trigger async operations and see real-time status, progress, errors, and retry options directly in the UI.

## Frontend Changes

### Job API Client
- Typed Jobs API routes and client functions (`frontend/src/lib/api-client.ts`, `api-types.ts`, `jobs.ts`).
- Support for `GET /api/jobs`, `GET /api/jobs/{job_id}`, and `POST /api/jobs/{job_id}/retry`.
- Async sync endpoint: `POST /api/mailboxes/{mailbox_id}/sync-jobs`.
- Async digest endpoints: `POST /api/digest/today/generate-jobs` and `POST /api/digest/today/refresh-jobs`.

### Reusable Job UI Components
- `JobStatusBadge`: Visual status indicator (queued, running, completed, failed, cancelled).
- `JobProgressCard`: Progress display with `role="progressbar"` and numeric aria values.
- `JobErrorPanel`: Redacted error message display with error code.
- `RetryJobButton`: Failed job retry with native disabled semantics.
- `JobActivityList`: Recent jobs list with status, progress, and retry actions.

### Job Polling Hooks
- `useJobPolling`: Polls `GET /api/jobs/{job_id}` until terminal status or timeout.
- `useRecentJobs`: Fetches recent jobs with `GET /api/jobs?limit=8`.

### Async Mailbox Sync Experience
- `/settings/mailboxes`: Sync Today now creates async `email_sync` jobs via `POST /api/mailboxes/{mailbox_id}/sync-jobs`.
- Polls job status until completion.
- Refreshes mailbox status on completion.
- Shows retry option on failure.
- Falls back to synchronous `POST /api/mailboxes/{mailbox_id}/sync` if async creation fails.

### Async Digest Experience
- `/dashboard`: Generate Digest now creates async `digest_generate` jobs via `POST /api/digest/today/generate-jobs`.
- `/dashboard`: Refresh Digest now creates async `digest_refresh` jobs via `POST /api/digest/today/refresh-jobs`.
- Polls job status until completion.
- Loads digest by `digest_id` when available, or falls back to `GET /api/digest/today`.
- Falls back to synchronous endpoints if async creation fails.

### Recent Jobs / Background Activity
- `/actions`: Added Background Activity section showing recent jobs via `GET /api/jobs?limit=8`.
- Supports failed-job retry directly from the list.

### i18n Coverage
- Added English and Chinese job-experience text to `en.json` and `zh.json`.
- Covers job status, progress, errors, retry, and activity labels.

### Theme Compatibility
- Job components use existing MailMind tokens, cards, badges, feedback panels, and progress rails.
- No new theme preset or external UI dependency added.

### Accessibility
- Job progress uses `role="progressbar"` with numeric aria values.
- Retry buttons use native disabled semantics.

## Backend Changes

None. v0.4 is a frontend-only release that wires the existing v0.3 Jobs API.

## API Changes

No new API endpoints. v0.4 uses the existing v0.3 Jobs API:

- `GET /api/jobs` — list jobs with filters.
- `GET /api/jobs/{job_id}` — get job detail.
- `POST /api/jobs/{job_id}/retry` — retry failed job.
- `POST /api/mailboxes/{mailbox_id}/sync-jobs` — async email sync.
- `POST /api/digest/today/generate-jobs` — async digest generate.
- `POST /api/digest/today/refresh-jobs` — async digest refresh.

## Database / Migration Changes

No new schema migrations. v0.4 does not modify the database.

## Runtime / Worker Notes

- If the Celery worker is not running, jobs remain queued/running and the UI keeps showing the job card until terminal status or polling timeout.
- If async job creation fails, mailbox sync and digest actions fall back to existing synchronous endpoints.
- The frontend displays only backend-provided redacted `error_message` / `error_code`.
- No token, session, prompt, API key, or email body is written to localStorage.

## Security Notes

- Frontend displays only backend-redacted error messages.
- No sensitive data (tokens, keys, prompts, email bodies) is stored in localStorage.
- Job polling respects user ownership (backend enforces `user_id` checks).

## Known Limitations

- Scheduled jobs still require manual/external trigger unless a production scheduler is later added.
- AI Provider settings remain env-based; no in-app UI for provider management.
- Multi Mailbox is not included.
- Outlook / IMAP are not included.
- Production deployment is not included.
- `GET /api/emails/new` remains planned but not implemented.
- Backend job `result` may not always include `digest_id` for completed digest jobs; frontend falls back to `GET /api/digest/today`.

## Next Roadmap

- v0.5 Open Source Ready: CI, Docker polish, public docs review.
- v1.0 Personal Productivity Ready: stabilize Daily Digest for daily personal use.
- Future: Multi Mailbox, Outlook / IMAP, in-app AI provider settings.
